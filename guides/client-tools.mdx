zot exposes a standard OCI Distribution API, so any conformant client works out of the box. This guide covers:

- **zli** — the official zot CLI for querying images, CVEs, repos, and server info
- **zb** — the zot benchmarking tool
- **Docker CLI**, **skopeo**, and **crane** for push/pull operations

---

## zli

`zli` is a command-line client built specifically for zot. It queries zot's extended GraphQL API to provide rich information about images and CVEs that standard OCI clients do not expose.

<Note>
  `zli` requires the **search** extension to be enabled on the zot server (`extensions.search.enable: true`). Most subcommands use the GraphQL endpoint and will return an error if the extension is unavailable.
</Note>

### Installation

Download a pre-built binary from the [zot releases page](https://github.com/project-zot/zot/releases) and place it on your `PATH`:

```bash
# Linux amd64
curl -Lo zli https://github.com/project-zot/zot/releases/latest/download/zli-linux-amd64
chmod +x zli
sudo mv zli /usr/local/bin/zli
```

Verify the installation:

```bash
zli --version
```

### Configuration

Before running commands, register your registry server with `zli config add`. This stores the URL and default options in `~/.zot`.

```bash
zli config add main https://zot.example.com:8080
```

List all configured registries:

```bash
zli config --list
```

View or change settings for a specific registry:

```bash
# Show all settings for "main"
zli config main --list

# Read a single setting
zli config main url

# Disable TLS verification (development only)
zli config main verify-tls false

# Hide the spinner when loading data
zli config main showspinner false
```

Remove a registry:

```bash
zli config remove main
```

**Supported config variables:**

| Variable | Default | Description |
|---|---|---|
| `url` | — | zot server URL. |
| `showspinner` | `true` | Show a spinner while queries are running. |
| `verify-tls` | `true` | Verify the server's TLS certificate. |

You can also pass `--url` or `--config` flags directly to any command instead of using a stored config entry.

### Authentication

Pass credentials inline with `--user`:

```bash
zli image list --config main --user admin:secretpassword
```

## Image commands

`zli image` lists and inspects images hosted on the registry.

```
zli image [command]
```

**Global flags** (available on all `image` subcommands):

| Flag | Description |
|---|---|
| `--url` | Registry URL (alternative to `--config`). |
| `--config` | Name of the stored config entry. |
| `-u`, `--user` | Credentials in `username:password` format. |
| `-f`, `--format` | Output format: `text`, `json`, or `yaml`. |
| `--verbose` | Show extended output. |
| `--debug` | Show HTTP debug output. |

### list — list all images

```bash
zli image list --config main
```

Example output:

```
REPOSITORY                TAG       DIGEST    CONFIG    LAYERS SIZE  LAST UPDATED
ubuntu                    22.04     a8652db5  23bab0be  1      28MB  2024-01-10 12:00:00
nginx                     1.25      e3c7d4f1  a1b2c3d4  5      67MB  2024-01-09 08:30:00
myapp/backend             v2.1.0    9f3a1b2c  f4e5d6a7  8      142MB 2024-01-08 15:45:00
```

Sort the output:

```bash
zli image list --config main --sort-by alpha-asc
```

### name — show a specific image

```bash
zli image name myapp/backend:v2.1.0 --config main
```

### cve — list CVEs for an image

List all CVEs affecting a specific image tag:

```bash
zli image cve myapp/backend:v2.1.0 --config main
```

Search for a specific CVE:

```bash
zli image cve myapp/backend:v2.1.0 --config main --cve-id CVE-2023-44487
```

### base — list base images

List the images that a given image is derived from:

```bash
zli image base myapp/backend:v2.1.0 --config main
```

### derived — list derived images

List all images that use a given image as a base:

```bash
zli image derived ubuntu:22.04 --config main
```

### digest — find images by digest

Find all images that contain a blob (manifest, config, or layer) with the given digest:

```bash
zli image digest sha256:8a1930f0 --config main
# Short digests are accepted
zli image digest 8a1930f0 --config main
```

## CVE commands

`zli cve` queries CVE data for images. The server must have the CVE scanner enabled (`extensions.search.cve`).

```
zli cve [command]
```

### list — CVEs for an image

List all CVEs for an image by tag or digest:

```bash
zli cve list myapp/backend:v2.1.0 --config main
```

Example output:

```
ID              SEVERITY  TITLE
CVE-2023-44487  HIGH      HTTP/2 Rapid Reset Attack
CVE-2023-38545  CRITICAL  curl SOCKS5 heap buffer overflow
CVE-2023-29491  MEDIUM    ncurses memory corruption
```

Sort by severity (default) or alphabetically:

```bash
zli cve list myapp/backend:v2.1.0 --config main --sort-by severity
```

### affected — images affected by a CVE

Find all images in the registry that are affected by a specific CVE:

```bash
zli cve affected CVE-2023-44487 --config main
```

Scope the search to a specific repository:

```bash
zli cve affected CVE-2023-44487 --config main --repo myapp/backend
```

### fixed — tags where a CVE is resolved

Find which tags of a repository have a fix for a given CVE:

```bash
zli cve fixed myapp/backend CVE-2023-44487 --config main
```

This is useful for identifying the minimum tag you can upgrade to in order to eliminate a vulnerability.

### diff — CVE difference between two images

List CVEs that are present in the first image but not in the second (useful for comparing two tags or upgrade candidates):

```bash
zli cve diff myapp/backend:v2.0.0 myapp/backend:v2.1.0 --config main
```

For multi-arch images, specify the platform:

```bash
zli cve diff myapp/backend:v2.0.0 linux/amd64 myapp/backend:v2.1.0 linux/amd64 --config main
```

## Repo commands

`zli repo` lists repositories available in the registry.

### list — list all repositories

```bash
zli repo list --config main
```

Example output:

```
NAME
myapp/backend
myapp/frontend
ubuntu
nginx
```

## Search commands

`zli search` performs fuzzy search across repository names and tags using the GraphQL search index.

### query — fuzzy search

Search for repositories by substring:

```bash
zli search query "myapp" --config main
```

Search for images by repo name and partial tag:

```bash
zli search query "myapp/backend:v2." --config main
```

List all images (use `:` as the query):

```bash
zli search query ":" --config main
```

### subject — list referrers

List all OCI referrers attached to an image (signatures, SBOMs, attestations):

```bash
zli search subject myapp/backend:v2.1.0 --config main
# or by digest
zli search subject "myapp/backend@sha256:9f3a1b2c..." --config main
```

## Server info command

`zli status` checks whether the server is reachable and reports its version and build information.

```bash
zli status --config main
```

Example output:

```
Server Status: online
Server Version: v2.1.0
Dist Spec Version: 1.1.1
Built with: search, sync, metrics, lint
```

Output in JSON:

```bash
zli status --config main --format json
```

```json
{
    "status": "online",
    "distSpecVersion": "1.1.1",
    "commit": "a3f2b1c",
    "binaryType": "-search-sync-metrics",
    "releaseTag": "v2.1.0"
}
```

## zb — benchmarking tool

`zb` is a load-testing and benchmarking tool for any OCI Distribution-conformant registry, including zot. It runs a suite of push and pull tests and reports latency percentiles and throughput.

### Installation

```bash
# Linux amd64
curl -Lo zb https://github.com/project-zot/zot/releases/latest/download/zb-linux-amd64
chmod +x zb
sudo mv zb /usr/local/bin/zb

# Run with Docker (no installation)
docker run --net=host ghcr.io/project-zot/zb-linux-amd64:latest \
  -c 2 -n 10 http://localhost:5000
```

### Usage

```
zb <url> [flags]
```

| Flag | Short | Description |
|---|---|---|
| `--auth-creds` | `-A` | Basic auth credentials in `user:pass` format. |
| `--concurrency` | `-c` | Number of concurrent requests (default: 1). |
| `--requests` | `-n` | Total number of requests to perform (default: 1). |
| `--repo` | `-r` | Repository name on the registry used for test data. |
| `--working-dir` | `-d` | Local directory used to store generated test data. |
| `--src-cidr` | `-s` | CIDR range to source requests from (mutually exclusive with `--src-ips`). |
| `--src-ips` | `-i` | Colon-separated source IPs (mutually exclusive with `--src-cidr`). |
| `--output-format` | `-o` | Output format: `stdout` (default), `json`, `ci-cd`. |
| `--test-regex` | `-t` | Run only tests whose name matches this regex. |
| `--list-tests` | `-l` | Print available test names and exit. |
| `--skip-cleanup` | — | Leave pushed repos on the registry after the run. |

## Docker CLI

zot is compatible with the Docker CLI for standard push and pull operations. Enable Docker Schema 2 compatibility by adding `"compat": ["docker2s2"]` to your HTTP configuration:

```json
{
    "distSpecVersion": "1.1.1",
    "storage": {
        "rootDirectory": "/tmp/zot"
    },
    "http": {
        "address": "127.0.0.1",
        "port": "8080",
        "compat": ["docker2s2"]
    },
    "log": {
        "level": "debug"
    }
}
```

With Docker compatibility enabled:

```bash
# Login
docker login localhost:8080

# Pull an image from Docker Hub and push to zot
docker pull ubuntu:22.04
docker tag ubuntu:22.04 localhost:8080/ubuntu:22.04
docker push localhost:8080/ubuntu:22.04

# Pull from zot
docker pull localhost:8080/ubuntu:22.04
```

<Note>
  Without `"compat": ["docker2s2"]`, zot accepts only OCI media types. If you push a Docker Schema 2 image without the compat flag, the push will succeed but the image may not be pullable by Docker clients that expect Docker Schema 2 responses.
</Note>

## skopeo and crane

<Tabs>
  <Tab title="skopeo">
    [skopeo](https://github.com/containers/skopeo) is a versatile tool for copying images between registries, inspecting manifests, and converting formats.

    Copy an image from Docker Hub to zot:

    ```bash
    skopeo copy \
      docker://ubuntu:22.04 \
      docker://localhost:8080/ubuntu:22.04
    ```

    Copy in OCI format (recommended for zot):

    ```bash
    skopeo copy \
      --format=oci \
      docker://ubuntu:22.04 \
      docker://localhost:8080/ubuntu:22.04
    ```

    Skip TLS verification for a local registry:

    ```bash
    skopeo copy \
      --format=oci \
      --dest-tls-verify=false \
      docker://gcr.io/google-samples/hello-app:1.0 \
      docker://localhost:5001/hello-app:1.0
    ```

    Inspect an image manifest without pulling layers:

    ```bash
    skopeo inspect docker://localhost:8080/ubuntu:22.04
    ```

    Copy between two zot registries:

    ```bash
    skopeo copy \
      docker://source-registry:8080/myapp:v1.0 \
      docker://dest-registry:8080/myapp:v1.0
    ```
  </Tab>

  <Tab title="crane">
    [crane](https://github.com/google/go-containerregistry/blob/main/cmd/crane/README.md) is a Go-based tool from the `go-containerregistry` project. It is lightweight and works well for scripting.

    Install:

    ```bash
    curl -sL https://github.com/google/go-containerregistry/releases/latest/download/go-containerregistry_Linux_x86_64.tar.gz \
      | tar -xzf - crane
    sudo mv crane /usr/local/bin/crane
    ```

    Copy an image:

    ```bash
    crane copy ubuntu:22.04 localhost:8080/ubuntu:22.04
    ```

    For insecure (HTTP) registries:

    ```bash
    crane copy --insecure ubuntu:22.04 localhost:8080/ubuntu:22.04
    ```

    List tags:

    ```bash
    crane ls localhost:8080/ubuntu
    ```

    Inspect a manifest:

    ```bash
    crane manifest localhost:8080/ubuntu:22.04
    ```

    Crane supports the `registryToken` field in `~/.docker/config.json`, which makes it suitable for OIDC bearer token authentication (e.g., Kubernetes workload identity). Write the token directly:

    ```json
    {
      "auths": {
        "registry.example.com:5000": {
          "registryToken": "<bearer-token>"
        }
      }
    }
    ```
  </Tab>
</Tabs>
